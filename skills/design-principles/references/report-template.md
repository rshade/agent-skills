# Report template

Template for DESIGN_AUDIT.md generated by the `design-principles` skill.

## Impact levels

| Impact | Definition | Example |
| ------ | ---------- | ------- |
| LOW | Minor design smell, low reader confusion | Single extra abstraction level |
| MEDIUM | Slows feature development or increases test burden | Mixed concerns in a handler |
| HIGH | Blocks testing, causes recurring bugs, or requires shotgun surgery | No interfaces at layer boundaries |
| CRITICAL | Causes production risk or blocks meaningful refactoring | Hardcoded credentials, untestable core logic |

## Effort levels

| Effort | Time | Example |
| ------ | ---- | ------- |
| S | < 1 day | Extract constant, add delegation method |
| M | 1–5 days | Introduce interface, split a God file |
| L | 1–2 weeks | Restructure a layer, add DI across a subsystem |
| XL | > 2 weeks | Rewrite a core module to untangle concerns |

## Priority calculation

| Priority | Criteria | Action |
| -------- | -------- | ------ |
| P1 | HIGH/CRITICAL impact + S/M effort | Fix this sprint |
| P2 | HIGH impact + L effort, or MEDIUM + S/M effort | Fix next sprint |
| P3 | MEDIUM impact + L effort | Fix this quarter |
| P4 | LOW impact, any effort | Backlog |

## Principle scores (0–10)

| Score | Level | Description |
| ----- | ----- | ----------- |
| 0–2 | Excellent | No significant violations found |
| 3–5 | Good | Minor violations, not blocking progress |
| 6–8 | Needs work | Multiple violations slowing development |
| 9–10 | Critical | Pervasive violations, blocking refactoring |

Score each principle on how severely its violations impact the
codebase, not just how many violations exist. One CRITICAL finding
scores higher than five LOW findings.
